You are analyzing cloud infrastructure defined in Terraform/OpenTofu to generate a structured data flow diagram.

Based on the infrastructure analysis provided, extract and organize the components and data flows into a structured JSON format that will be used to generate a Data Flow Diagram (DFD).

# Component Categories

Categorize each infrastructure component into one of these types:

1. **tenant** - Cloud account/subscription level (AWS Account, GCP Project, Azure Subscription, OCI Tenancy)
2. **container** - Resource grouping (Azure Resource Group, OCI Compartment)
3. **network** - Network boundaries (AWS VPC, VCN, Subnet, Route Table)
4. **network_access_control** - Network access control mechanisms (AWS Security Group, Network ACL, firewall rules/policies, firewalls, web application firewalls/cloud armor, GCP VPC service controls)
5. **gateway** - Network gateways (Internet Gateway, NAT Gateway, VPN Gateway, Load Balancers, peering connections, VPC endpoints, private endpoints/privatelink/service connect, direct connect)
6. **compute** - Compute resources (EC2, VM, Container Instance, Function)
7. **service** - Logical groupings of compute units
8. **storage** - Data stores (RDS, S3, Blob Storage, Object Storage, Database)
9. **actor** - External entities (Users, External Systems, Internet, Third-party APIs)

# Hierarchy and Nesting

Understanding proper containment relationships is critical:

- **tenant** contains everything else
- **container** (if present) contains network, gateway, compute, and storage
- **network.vcn** contains network.subnet
- **network.subnet** contains compute and storage resources
- **network.security_group** applies to (but doesn't contain) compute/storage resources; it can be modeled as a 
- **gateway** resources sit at network boundaries
- **actor** resources are external and not contained by anything

## Layout Optimization Guidelines

To ensure clear, readable diagrams, follow these nesting principles:

1. **Minimize Nesting Depth**: Avoid deeply nested hierarchies that create visual stacking. Prefer flatter structures where possible.

2. **Sibling Components**: When multiple components of the same type exist (e.g., multiple subnets, multiple compute instances), they should be siblings (share the same parent_id) rather than nested within each other.

3. **Network Boundaries**:
   - Create **separate** components for logically distinct networks (public vs private, different availability zones)
   - Place compute/storage resources as children of their respective networks
   - Do NOT nest subnets within security groups - security groups should be a peer object of the object they apply to, and flows should route through the network security group to the object(s) they protect

4. **Logical Grouping**: Group components by their functional tier or layer:
   - Public tier: Load balancers, bastion hosts (in public subnets)
   - Application tier: Web servers, application servers (in private subnets)
   - Data tier: Databases, storage (in private/database subnets)
   - Each tier's resources should be siblings within their respective subnet containers

5. **Gateway Placement**: Gateways (IGW, NAT Gateway, VPN Gateway) should typically be children of the VPC/VCN, not nested inside subnets, as they serve the entire network.

6. **Nesting Objects**: objects nested inside other objects, should be positioned and sized to fit completely inside the parent object.  A child object should never overlap the boundary of its parent, or be positioned outside that boundary.

7. **No Overlap**: objects should not overlap other objects unless nested inside the other object.

8. **Peer Object Positioning**: Peer objects should be laid out in a grid horizontally and vertically inside the object that contains them, without overlapping.  This applies to all kinds of objects including container objects.  Top level objects should be organized in a grid in the graph.

**Example of GOOD structure (flat, clear layout):**
```
tenant
├── vpc
│   ├── internet-gateway
│   ├── nat-gateway
│   ├── subnet-public-az1
│   │   ├── load-balancer
│   │   ├── bastion-host
│   ├── subnet-private-az1
│   │   ├── web-server-1
│   │   ├── web-server-2
│   ├── subnet-database-az1
│   │   ├── rds-primary
├── actor-internet
```

**Example of BAD structure (too much nesting, stacking):**
```
tenant
├── vpc
│   ├── subnet-public
│   │   ├── security-group-lb
│   │   │   ├── load-balancer
│   │   │   │   ├── security-group-web
│   │   │   │   │   ├── web-server
```

# Output Format

Generate a JSON object with two main sections: "components" and "flows".

## Components

Each component should include:
- **id** - Unique identifier (use resource name or generate one)
- **name** - Display name for the diagram
- **type** - One of: tenant, container, network, gateway, compute, storage, actor
- **subtype** - Specific resource type (vpc, subnet, security_group, ec2, rds, etc.)
- **description** - Brief description of the resource
- **parent_id** - ID of the parent component (null for top-level tenant)
- **metadata** - Additional properties (region, availability_zone, tags, etc.)

## Flows

Each data flow should include:
- **id** - Unique identifier for the flow
- **name** - Description of the data flow
- **source_id** - Component ID where data originates
- **target_id** - Component ID where data goes
- **protocol** - Protocol used (HTTPS, HTTP, SSH, TLS, TCP, UDP, etc.)
- **port** - Port number if applicable
- **data_type** - Type of data being transferred (API requests, web traffic, database queries, etc.)
- **bidirectional** - Boolean indicating if this is a two-way communication

# Special Considerations

1. **Bidirectional Flows**: If communication goes both ways (e.g., HTTPS request/response), set "bidirectional": true. We will create two unidirectional edges from this.

2. **Network Boundaries**: Create subnet, and VCN components as separate boundary objects even if they don't directly participate in flows.

3. **Network Access Controls**: Create process objects for network access controls, and use the network access control node as a separate hop in flows that are subject to that control.  For example, if a process protected by a network access control communicates with a process that is not protected by the same NAC object, then flows should be created from the first process to the NAC object, and from the NAC object to the second process, and the same pattern applies in flows in the other direction.

4. **Load Balancers**: Treat as gateway components that receive flows from actors and send flows to compute resources.

5. **Gateways**: Create an "actor" component with an appropriate name to represent what is on the opposite side of a gateway, for example, for internet gateways, create an "actor" called "Internet".

5. **Database Access**: Storage components (RDS, databases) should receive flows from compute components that write to them and and should have flows to compute components that read from them.

6. **Implicit Flows**: Infer flows even if not explicitly stated. For example:
   - Web servers behind load balancers receive traffic from the load balancer
   - Application servers query databases
   - Instances in private subnets route through NAT gateways for outbound internet
