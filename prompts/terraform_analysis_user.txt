**IMPORTANT: Mermaid Syntax Rules (you MUST follow these exactly):**
- Every `subgraph` statement MUST have a corresponding `end` statement
- Do NOT use `<br/>` or `<br>` tags - they are not supported. Use short labels instead
- Do NOT use ampersands (`&`) - use the word "and" instead
- Do NOT use special characters in node labels except underscores and hyphens
- Keep node labels short and simple (under 40 characters)
- Use only alphanumeric characters, underscores, and hyphens for node IDs

Use this format:
```mermaid
graph TD
    %% Define nodes
    Internet[Internet]
    LB[Load Balancer]

    %% Subgraphs must have matching end statements
    subgraph VPC[Virtual Private Cloud]
        AppServers[Application Servers]
        DB[Database]
    end

    %% Define relationships
    Internet -->|HTTPS| LB
    LB -->|HTTP| AppServers
    AppServers -->|SQL| DB

    %% Style important nodes
    classDef publicFacing fill:#ff9999
    classDef sensitive fill:#ffcc99
    class Internet,LB publicFacing
```

Keep the diagram clear and focused on the most important components and flows. Use subgraphs to represent security zones or boundaries when appropriate.

#### Services - Logical compute groupings

1. **Identify Logical Services**: A "service" is defined as a cohesive, logical grouping of compute resources that function collectively as a single entity, such as a scalable web tier, database cluster, or containerized application. Services simplify representation by aggregating individual compute units (e.g., EC2 instances, ECS tasks) and their supporting objects (e.g., load balancers, auto-scaling groups). Infer services based on the following criteria, prioritizing strong relational evidence:

   - **Resource Associations and Dependencies**: Group resources linked by references, such as ARNs, IDs, or explicit dependencies (e.g., an Auto Scaling Group referencing a Launch Template; instances registered to a Target Group; ECS services tied to tasks and clusters).
   - **Shared Attributes**: Aggregate by matching tags (e.g., "app:frontend", "env:prod"), metadata, or labels that indicate common ownership or purpose.
   - **Naming Patterns**: Cluster resources with consistent naming conventions (e.g., "api-asg" and "api-lb" suggest an "api-service").
   - **Functional Collaboration**: Identify groupings where resource types collaborate for a workload (e.g., EC2 instances + Load Balancer = load-balanced service; EKS node groups + Pods = Kubernetes service).
   - **Network and Security Boundaries**: Group by shared VPCs, subnets, security groups, or IAM roles, implying isolation within a service.
   - **Scaling and Resilience Features**: Prioritize entities with auto-scaling, health checks, or desired capacities, as these denote dynamic collectives.
   - **Data-Source Specific Rules**:
     - For Terraform: Use module structures, resource blocks, "depends_on" attributes, variables, and outputs to trace hierarchies (e.g., a module containing aws_autoscaling_group and aws_lb defines a service).
     - For JSON API Data: Correlate across objects using nested arrays, shared keys (e.g., InstanceIds in ASGs), or linked fields (e.g., LoadBalancerArns); treat top-level entities like AutoScalingGroups or Services as service candidates.
   - **Exclusion and Ambiguity Handling**: Do not group unrelated or standalone resources (e.g., isolated monitoring alarms). If links are weak or absent, default to singleton services. Resolve ambiguities by favoring larger, more interconnected groups to minimize fragmentation.

2. **Document Criteria for Each Service**: For every identified service, explicitly list the specific criteria (from the above) that justified its definition, including evidence from the data (e.g., "Shared tag 'app:web' on ASG and LB; ASG references LB ARN").

3. **List Associated Components**:

   - **Compute Units**: Enumerate the primary compute resources (e.g., EC2 instances, ECS tasks, EKS nodes) within the service, including counts or IDs where available.
   - **Other Objects**: Detail supporting resources (e.g., load balancers, clusters, launch configurations, security groups, subnets), with their identifiers.

4. **Output**: include the following information

   - **name**: Inferred or descriptive name (e.g., 'web-frontend') of the service
   - **criteria**: Bullet-point list of criteria used to map compute units to the service, with evidence
   - **compute units**: List of compute resource types and details (e.g., 'EC2 instances: 3 in ASG web-asg') that are part of the service
   - **associated objects**: List of non-compute resource types and details that are associated with the service, e.g. load balancers

Ensure the output is comprehensive yet concise, covering all relevant data without inventing information. If the input lacks sufficient links, note limitations in the output.
